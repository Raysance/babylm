Task Objective

任务目标简介

ZhoBLiMP：语言学最小对 zero-shot 评估

  • 评估对象：Decoder causal LM。
  • 评估形式：给定一对只在某个语言学现象上不同的句子，模型需要给可接受句子更高概率。
  • 覆盖现象：把字句、被动、照应、量词、疑问、关系从句、话题化、NPI/FCI 许可等。
  • 指标：average accuracy、field accuracy、UID accuracy。
  • 意义：无需 fine-tuning，直接检验预训练模型是否学到中文语法偏好。

CLUE-style fine-tuning：下游理解任务评估

Task评测内容模型需要做什么
AFQMC蚂蚁金融语义相似度判断两个问题是不是同一个意思。
OCNLI中文自然语言推理判断两句话是能推出、相矛盾，还是关系不确定。
TNEWS新闻标题分类根据标题判断新闻主题类别。
CLUEWSC2020中文代词指代判断句子里的“他/她/它”到底指谁。
Metric直观含义
Accuracy有多少题答对了，是最直观的正确率。
F1看模型有没有真正找出“是 / 相似 / 同指”这一类，避免只看总体答对率。
MCC看模型是不是在认真区分两类；如果总是猜同一个答案，MCC 会很低。
因此，本项目用 ZhoBLiMP 衡量语言形式偏好，用 CLUE 衡量下游任务迁移；两者共同构成 NLU track 的主要评估视角。

模型层级设计点

  • 共享词表：Decoder 与 Encoder 使用同一子词空间，避免词表差异影响对比。
  • 中文适配：SentencePiece 不依赖空格分词，适合中文语料。
  • 特殊 token 固定：使 causal LM 与 MLM 的 collate 逻辑都可稳定识别 padding、mask 和边界。
这不是新的 tokenizer 算法，但属于模型输入层面的统一设计，对双模型对比实验非常关键。

核心代码

def causal_lm_collate(batch):
    input_ids = [item["input_ids"][:MAX_LEN] for item in batch]
    input_ids = pad_sequence(input_ids, batch_first=True, padding_value=0)
    attention_mask = input_ids.ne(0).long()
    labels = input_ids.clone()
    labels[input_ids == 0] = -100
    return {"input_ids": input_ids, "attention_mask": attention_mask, "labels": labels}

模型层级设计点

  • 用 causal objective 对齐 ZhoBLiMP 的句子似然比较。
  • 紧凑 6-layer 配置适配 BabyLM 小语料与课程算力限制。
  • 保留 checkpoint sweep，使模型选择基于语言学评估，而非默认 final。

核心 mask 代码

probability_matrix = torch.full(labels.shape, MLM_PROBABILITY)
special_tokens_mask = (
    input_ids.eq(PAD_TOKEN_ID)
    | input_ids.eq(UNK_TOKEN_ID)
    | input_ids.eq(BOS_TOKEN_ID)
    | input_ids.eq(EOS_TOKEN_ID)
)
probability_matrix.masked_fill_(special_tokens_mask, 0.0)
masked_indices = torch.bernoulli(probability_matrix).bool()
labels[~masked_indices] = -100

模型层级设计点

  • 用 bidirectional MLM 对齐 CLUE 分类任务的句子表示需求。
  • 显式排除特殊 token，避免模型学习无意义的边界/填充预测。
  • 用紧凑 BERT-style backbone 支持受限数据下的稳定 fine-tuning。
Model-Level Design

模型层级设计：双 backbone 与任务适配

双 backbone 设计

  • Decoder 负责左到右句子概率建模，天然适配 minimal-pair scoring。
  • Encoder 负责双向上下文表示学习，天然适配句子分类和句对分类。
  • 两类 backbone 在同一数据和 tokenizer 条件下训练，便于比较预训练目标差异。

模型配置取舍

  • 小模型容量：6 layers、hidden size 384、12 heads，适配受限数据和有限算力。
  • 共享输入空间：统一 SentencePiece tokenizer，减少 tokenizer 差异带来的混杂因素。
  • 目标函数适配：causal LM 对齐 ZhoBLiMP；MLM 对齐 CLUE fine-tuning。
  • checkpoint 选择：Decoder 按 ZhoBLiMP 曲线选择最优 checkpoint，而不是默认 final。
Training State Module

Checkpoint 与续训模块运行流程

保存逻辑

def save_training_checkpoint(model, optimizer, scheduler, output_dir, global_step):
    save_model_and_tokenizer(model, TOKENIZER_PATH, output_dir)
    torch.save(optimizer.state_dict(), os.path.join(output_dir, "optimizer.pt"))
    torch.save(scheduler.state_dict(), os.path.join(output_dir, "scheduler.pt"))
    save_trainer_state(output_dir, global_step)

续训规则

  1. 如果 final/trainer_state.json 存在，从 final 继续。
  2. 否则从最新 checkpoint-* 继续。
  3. checkpoint 保存按 global_step % SAVE_STEPS == 0 判断。
例：在 step 163200 中断，下次继续后下一个 checkpoint 仍是 step 175000。

Retrospective

提升空间和分析复盘

当前问题

  • Decoder 最佳 checkpoint 不是 final，说明训练后期可能出现评估回落。
  • Encoder 在 AFQMC 上可能经常猜同一类，所以正确率看起来还行，但细看并不稳定。
  • 历史 encoder checkpoint loss 没有完整持久记录，影响训练过程分析。

后续改进

  • 增强指代建模能力，重点改善 anaphor 和 CLUEWSC2020 中“代词指谁”的判断。
  • 强化被动句和长距离依赖学习，减少 passive 相关结构的系统性错误。
  • 为 AFQMC 加强句对语义匹配能力，避免模型只依赖表面词重合或固定类别偏好。
  • 提高 encoder 表示的稳定性，让同义改写、推理关系和主题分类都能获得更一致的句子表示。
